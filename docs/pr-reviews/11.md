# PR Review Log #11

## Summary
- **Scope**: Add `chunk_id` column to messages table and `dedupe_by_session` mode in `search_messages()` to fix BM25 scoring — prevents a single session with many matching messages from crowding out other sessions in ranked search results.
- **Risk**: Medium (touches search/write path, schema migration v5 → v6)
- **Positives**:
  - Clean separation of dedupe logic from non-dedupe path
  - Good use of SQL window functions (ROW_NUMBER) for deduplication
  - Solid migration backfill with correlated subquery
  - Comprehensive test coverage (8 new tests: chunk_id format, uniqueness, search results, migration backfill, dedupe behavior)
  - Correct handling of FTS5 `snippet()` context limitation (must be in direct FTS query, not subquery)
- **Concerns**:
  1. Missing `idx_messages_chunk_id` index on fresh databases (fixed in follow-up commit)
  2. Silent error swallowing in `except sqlite3.OperationalError` (fixed in follow-up commit)
  3. Double FTS5 MATCH scan in dedupe path (minor perf, acceptable for typical workloads)
  4. Single-process assumption for chunk_id uniqueness (documented)

## Decision
Request changes — two major issues identified and fixed in follow-up commit:
1. Added `idx_messages_chunk_id` index creation post-migration (same pattern as `idx_sessions_title_unique`)
2. Narrowed `OperationalError` catch to FTS5-specific errors only

## Next steps
1. Benchmark dedupe query on production-scale data (100k+ messages)
2. Consider documenting single-process assumption for chunk_id generation
3. Monitor for edge cases in FTS5 error string matching
